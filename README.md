# Medical Clinic Console MVP

Консольное приложение для управления записями на приём к врачам.
Проект создан как демонстрация базовых навыков C# и принципов ООП.

## Архитектура

Проект разделён на три слоя:

- **`Interfaces/`** — контракты. `IClinic`.
- **`Services/`** — доменные модели (`Doctor`, `Patient`, `Appointment`) и бизнес-логика (`Clinic`).
- **`Program.cs`** — точка входа: настройка DI-контейнера и сценарий использования.

### Зачем интерфейс

`Program.cs` работает с `IClinic`, а не с `Clinic` напрямую. Это значит, что вызывающий код не знает деталей реализации. Если завтра появится `ClinicWithDatabase` — `Program.cs` не изменится ни строкой, поменяется только регистрация в контейнере.

### Зачем DI-контейнер

Раньше объект создавался напрямую: `IClinic clinic = new Clinic();` — `Program.cs` был жёстко привязан к конкретному классу.

Сейчас зависимость резолвится через `Microsoft.Extensions.DependencyInjection`:

```csharp
var services = new ServiceCollection();
services.AddSingleton<IClinic, Clinic>();
var provider = services.BuildServiceProvider();

var clinic = provider.GetRequiredService<IClinic>();
```

`Clinic` зарегистрирован как `Singleton` — он хранит список докторов и записей в памяти, и должен жить один экземпляр на всё время работы приложения.

Это даёт два конкретных преимущества:
- **Замена реализации** без изменения `Program.cs` — достаточно поменять регистрацию в контейнере.
- **Тестируемость** — в юнит-тестах можно подставить мок `IClinic` вместо реального `Clinic`, не трогая базу или состояние.

## Что реализовано

- **Модели данных:** `Doctor`, `Patient`, `Appointment` с валидацией через исключения
- **Бизнес-логика:** бронирование и отмена записей, защита от повторного бронирования, обработка несуществующих ID
- **DI-контейнер:** разрешение зависимостей через `ServiceCollection` вместо прямого `new`
- **Разделение ответственности:** сервисный слой бросает исключения, обработка — на стороне вызывающего кода
